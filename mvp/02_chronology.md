## Резюме проекта

**InfoTech.io** - интерактивная образовательная платформа с открытым контентом, построенная по архитектуре "ось и спицы". Проект реализован как федерация независимых статических учебных модулей, объединенных центральным хабом с полной автоматизацией CI/CD.

**Ключевые характеристики Архитектуры v2:**
- 🌐 Домен: infotecha.ru + wildcard поддомены
- 🏢 GitHub организация: info-tech-io (11 репозиториев)
- ⚙️ Технологии: Hugo Template Factory, Quiz Engine, Docker, GitHub Actions
- 📚 Готовые модули: 3 Linux курса + 1 Bash (готовы к сборке)
- 🏗️ Архитектура: Децентрализованная, управляемая `module.json`

## Этап 10: Первая попытка корпоративного портала и осознание проблемы инкрементной сборки

### 📅 **Временные рамки**

**Завершение:** 29 сентября 2025 (незавершённый этап — перешёл в новый эпик)
**Продолжительность:** ~8 дней (параллельно с Этапом 9)

### ⏱️ **Трудозатраты**
- **Факт:** ~40 часов

### 🔨 **Проделанная работа**

22 сентября основан репозиторий `info-tech-io.github.io` — будущий корпоративный портал организации и точка сборки всей документации. Замысел был амбициозным: единый сайт, объединяющий корпоративные страницы InfoTech.io и документацию всех дочерних продуктов (`quiz`, `hugo-templates`, `web-terminal`, `info-tech-cli`) в единой федерации по адресу `info-tech-io.github.io`.

Первый подход — монолитный workflow `deploy-sites.yml` (~316 строк), умевший собирать всё и сразу по одному триггеру. Следующие несколько дней — интенсивная отладка: проблемы с токенами, версией Hugo, сборкой через `hugo-templates`, git-субмодулями. 26 сентября корпоративный сайт впервые успешно задеплоился (`Deploy InfoTech.io Corporate Site - Etap 4 Complete`).

Однако попытки собрать **полную федерацию** — корпоративный сайт плюс документацию всех четырёх продуктов — раз за разом заканчивались неудачей. Проблема стала очевидной: монолитный подход работает только по принципу «всё или ничего». Пересобрать документацию только одного продукта при его обновлении было невозможно — workflow не умел работать инкрементно.

29 сентября принято стратегическое решение: **отключить все workflow** и начать проектирование архитектуры заново. В тот же день написан дизайн нового подхода и запущен **Epic #2: Rebuild GitHub Pages Federation with Incremental Architecture**.

### 💡 **Ключевое осознание**
Задача «собрать несколько независимых сайтов в один» оказалась нетривиальной. GitHub Actions в монолитном режиме не позволял пересобирать только изменившийся компонент: обновление документации Quiz Engine запускало полную пересборку всего портала. Нужна была архитектура, где каждый продукт управляет своей документацией независимо, а портал умеет принимать точечные обновления и встраивать их без пересборки остального. Это требование и стало основой для всей последующей работы.

## Этап 11: Epic #2 — Проектирование федеративной архитектуры (info-tech-io.github.io)

### 📅 **Временные рамки**
**Начало:** 29 сентября 2025
**Продолжительность:** ~5 дней (параллельно с началом Этапа 12)

### ⏱️ **Трудозатраты**
- **Факт:** ~20 часов

### 🔨 **Проделанная работа**

После стратегического решения отключить все workflow и начать заново, открыт **Epic #2: Rebuild GitHub Pages Federation with Incremental Architecture** в репозитории `info-tech-io.github.io`. Первый child — Investigation & Design: детальный анализ текущего состояния, инвентаризация всех workflow, выявление архитектурных ограничений.

В ходе исследования стало окончательно ясно: задача инкрементной сборки не решается на уровне workflow-файлов. Существующий `hugo-templates` умеет собирать один сайт — но не умеет оркестрировать несколько независимых сборок и сливать их в единую федерацию без полной пересборки. Нужен **второй слой логики**: orchestration-скрипт, который читает конфигурацию федерации, вызывает существующий `build.sh` для каждого продукта по отдельности, исправляет CSS-пути для поддиректорий и мёрджит результаты атомарно.

Child #2 Epic #2 назывался "Hugo Enhancement" — и оказался настолько крупным, что **превратился в отдельный эпик**: **Epic #15: Federated Build System** в репозитории `hugo-templates`. Работа по Epic #2 была поставлена на паузу до завершения инструментария.

### 💡 **Ключевое решение**
Не пытаться обойти ограничения существующих инструментов — а расширить сам инструмент. Добавить в `hugo-templates` полноценный второй уровень: федеративную оркестрацию поверх уже надёжного и протестированного одиночного build.

## Этап 12: Epic #15 — Строительство федеративного движка (hugo-templates)

### 📅 **Временные рамки**
**Начало:** 30 сентября 2025
**Завершение:** 26 октября 2025
**Продолжительность:** ~4 недели

### ⏱️ **Трудозатраты**
- **Факт:** ~120 часов

### 🔨 **Проделанная работа**

30 сентября в `hugo-templates` создан полный дизайн Epic #15. Архитектурное решение — **двухуровневая система** с принципом "zero breaking changes":
- Layer 1 (существующий, без изменений): `build.sh` + `module.json` — собирает один сайт
- Layer 2 (новый): `federated-build.sh` + `modules.json` — оркестрирует сборку нескольких сайтов в единую федерацию

Шесть дочерних задач выполнялись последовательно, каждая на отдельной feature-ветке с PR в epic-ветку:

- **Child #16 — Federated Build Script Foundation** *(1–6 октября):* Создан `scripts/federated-build.sh` — 1 149 строк, 22 функции. Парсинг `modules.json`, оркестрация вызовов `build.sh`, управление временными директориями, слияние выходных данных.

- **Child #17 — Modules.json Schema** *(6–7 октября):* JSON Schema (298 строк) для валидации конфигурации федерации, inline-валидатор на Bash без внешних зависимостей, 16 тестов — 100% pass rate, пользовательская документация (1 077 строк).

- **Child #18 — CSS Path Resolution** *(7–9 октября):* Решена нетривиальная проблема: Hugo генерирует абсолютные пути (`/css/style.css`), которые ломаются при размещении сайта в поддиректории (`/docs/quiz/`). Реализованы функции сканирования и переписывания путей в HTML-файлах. 10 тестов, все проходят.

- **Child #19 — Download-Merge-Deploy** *(13–17 октября):* Ключевой компонент инкрементности. Функция `download_existing_pages()` скачивает актуальное состояние GitHub Pages перед деплоем. Флаг `--preserve-base-site` сохраняет существующий контент и добавляет только изменившийся продукт — без пересборки остальных. Четыре стратегии слияния конфликтов.

- **Child #20 — Testing Infrastructure** *(18–19 октября):* **140 тестов, 100% passing.** Покрытие Layer 1 + Layer 2: unit-тесты, интеграционные тесты E2E, бенчмарки производительности. Производительность оказалась в разы лучше ожиданий: 5 модулей собираются за ~1.3 секунды (цель была < 60 секунд).

- **Child #21 — Documentation & Migration** *(20 октября):* **5 949 строк документации** за 4.5 часа (план был 12 часов). README, user guides, tutorials, API reference, migration guide. Epic #15 закрыт.

Параллельно, в ходе работы над Epic #15, в `hugo-templates` закрыты два технических бага: Issue #14 (кеш блокировал кастомный контент) и Issue #22 (build.sh возвращал exit code 1 при успешной сборке).

**25–26 октября** — после слияния epic-ветки в main выявлена и исправлена серия интеграционных проблем: ошибки определения кода возврата `git clone`, синтаксис `sed`, формат параметров `build.sh`. Это было первое боевое применение системы.

### 🔄 **Архитектурные изменения**
- **`hugo-templates` v0.3.0:** Фреймворк получил второй уровень — федеративную оркестрацию. Теперь одна команда собирает и деплоит несколько независимых сайтов как единую федерацию.
- **`--preserve-base-site`:** Флаг, решивший исходную проблему — инкрементное обновление одного продукта без пересборки всей федерации.

## Этап 14: Стабилизация — гонка потоков, баги и итеративные исправления

### 📅 **Временные рамки**
**Начало:** 28 октября 2025
**Завершение:** 6 ноября 2025
**Продолжительность:** ~10 дней

### ⏱️ **Трудозатраты**
- **Факт:** ~25 часов

### 🔨 **Проделанная работа**

**28 октября** — Child #5 (Integration Testing) сразу же дал результат. Уже в Stage 1 (Environment Setup) было обнаружено **критическое архитектурное уязвимое место**: гонка потоков между двумя workflow-файлами. Оба workflow (`deploy-corporate-incremental.yml` и `deploy-docs-federation.yml`) читали текущее состояние GitHub Pages, каждый модифицировал его независимо и деплоил — и второй deployment просто затирал результаты первого. Разные concurrency-группы (`pages` и `pages-docs`) позволяли им работать параллельно при одной целевой среде.

Решение было принято незамедлительно и радикально: вместо фикса concurrency — **полная архитектурная реорганизация**. Создан единый `deploy-github-pages.yml`: один атомарный workflow, одна concurrency-группа, одна транзакция сборки и деплоя. Параллельно удалено **16 устаревших и отключённых workflow-файлов**, накопившихся со времён первых экспериментов. Итог: архитектурная проблема устранена полностью.

**1 ноября** — **Issue #9: Fix Documentation Build Errors** открыт и закрыт в тот же день, за 3.5 часа. Причина: в двух репозиториях (`hugo-templates` и `info-tech-cli`) обнаружены bitrot-баги в content-файлах, не проявлявшиеся в обычных сборках, но ломавшие production-деплой через federation workflow. Исправлено 4 файла в двух репозиториях. После фикса все четыре документационных сайта вернули HTTP 200 OK.

**3 ноября** — **Issue #10: Fix Incremental Deployment Logic**, Stage 2. Выявлена вторая проблема: workflow с флагом `--preserve-base-site` пытался загрузить базовый сайт через `wget https://info-tech-io.github.io`, получал rate-limiting ошибки и падал. При этом актуальное состояние сайта **уже было скачано** в предыдущей фазе того же workflow. Решение — передать путь к уже загруженной директории (`--base-site-path=../current-site`) вместо URL. Stage 2 выполнен за 4 часа, кастомная merge-логика заменена вызовом `federated-build.sh`. Stages 3–4 (Enhanced Safety + Testing) остались незакрытыми и переносятся вперёд.

### 💡 **Ключевое наблюдение**
Этот период — естественная фаза "первого боевого применения". Каждый новый сложный инструмент (federated build system) неизбежно проходит через волну пост-релизных багов, которые невозможно предугадать до реальной интеграции. Три проблемы, найденные за 10 дней, были принципиально разными по природе: архитектурная (race condition), контентная (bitrot в markdown), и интеграционная (неверная передача параметров). Всё это — признак живой, работающей системы.

### 🔄 **Архитектурные изменения**
- **Атомарный unified workflow:** `deploy-github-pages.yml` заменил два параллельных workflow, устранив возможность race condition на уровне архитектуры.
- **Чистка workflow-директории:** 16 устаревших файлов удалены — репозиторий стал существенно чище.

## Архитектурная эволюция проекта

### Уровень 1: Инфраструктура модулей (июль – сентябрь 2025)

**Исходная архитектура (MVP, до сентября):**
- `hugo-base` — жёсткий шаблон без конфигурируемости
- `modules.json` — централизованный реестр, требующий ручного обновления хаба при добавлении модуля

**После Этапов 1–8 (Архитектура v2):**
- `hugo-templates` — параметрическая фабрика шаблонов с выбором из 4 вариантов
- `module.json` — децентрализованная конфигурация в каждом модуле: модуль сам описывает, как его собирать
- CI/CD: каждый `mod_*` репозиторий деплоится независимо, без изменений в хабе

### Уровень 2: Корпоративный портал и федерация (сентябрь – ноябрь 2025)

**Исходная проблема (Этапы 10–11):**
- Монолитный workflow: «всё или ничего» — обновление одного продукта требует полной пересборки всего портала
- Нет инкрементности — невозможно обновить документацию Quiz Engine, не тронув остальных

**После Этапов 12–13 (Архитектура v3):**
- `federated-build.sh` (Layer 2) — оркестрация нескольких независимых сборок в единую федерацию
- `--preserve-base-site` — инкрементное обновление: каждый продукт обновляет только свой раздел
- Repository Dispatch — каждый продукт при push сам уведомляет портал о необходимости точечной пересборки
- CSS-path rewriting — корректная работа статических ресурсов в поддиректориях (`/docs/quiz/`, `/docs/hugo-templates/`)

**После Этапов 14–15 (Стабилизация + Migration Readiness):**
- Атомарный unified workflow — устранена race condition через single concurrency group
- Epic #12 подготовлен: 21 зависимость инвентаризирована, staging environment готов, план миграции утверждён

### Итоговый технологический стек (ноябрь 2025)

```
info-tech-io.github.io (корпоративный портал)
  ├── deploy-github-pages.yml (единый атомарный workflow)
  │   ├── Corporate site ← info-tech/docs/ (через hugo-templates)
  │   └── Docs federation ← quiz, hugo-templates, web-terminal, info-tech-cli
  │       └── federated-build.sh (Epic #15)
  └── Repository Dispatch ← все продуктовые репозитории

infotecha.ru (образовательная платформа)
  ├── mod_linux_* (4 модуля контента)
  └── Полностью независима от GitHub Pages
```
