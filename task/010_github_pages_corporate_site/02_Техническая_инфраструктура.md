# Этап 2: Техническая инфраструктура и шаблоны

## Цель этапа
Расширить hugo-templates framework для поддержки корпоративных сайтов и документации, создать хаб-репозиторий для GitHub Pages и настроить базовые CI/CD процессы автоматизации.

## Задачи

### 2.1 Расширение hugo-templates новыми шаблонами (2 дня)
**Цель:** Добавить поддержку корпоративных сайтов и документации в существующий framework

**Действия:**
- Создать шаблон `corporate` для корпоративных сайтов организаций
- Создать шаблон `documentation` для технической документации продуктов
- Адаптировать существующую тему Compose для новых шаблонов
- Обновить build.sh скрипты для поддержки новых типов сайтов
- Создать примеры конфигурации module.json для каждого шаблона

**Техническая реализация:**
```
hugo-templates/templates/
├── educational/      # Существующий шаблон для образовательных модулей
├── corporate/        # Новый: для корпоративных сайтов
└── documentation/    # Новый: для документации продуктов
```

**Выходные артефакты:**
- Функциональный шаблон corporate с примером сборки
- Функциональный шаблон documentation с примером сборки
- Обновленные build.sh скрипты
- Примеры module.json для новых шаблонов

### 2.2 Создание хаб-репозитория GitHub Pages (1 день)
**Цель:** Подготовить центральный репозиторий для сборки и развертывания всех сайтов

**Действия:**
- Создать репозиторий info-tech-io.github.io
- Настроить GitHub Pages в репозитории
- Создать базовую структуру для размещения корпоративного сайта и документации
- Подготовить систему автоматической сборки с использованием hugo-templates
- Настроить PAT токены для доступа к репозиториям организации

**Структура хаб-репозитория:**
```
info-tech-io.github.io/
├── .github/workflows/
│   └── deploy-complete-sites.yml    # Основной workflow сборки
├── hugo-templates/                  # Субмодуль с фреймворком
├── build/                          # Временные файлы сборки
└── public/                         # Результат сборки (GitHub Pages)
```

**Выходные артефакты:**
- Настроенный хаб-репозиторий с GitHub Pages
- Работающий процесс автоматической сборки
- Базовая структура для размещения контента

### 2.3 Стандартизация структуры docs/ во всех продуктах (1 день)
**Цель:** Привести все репозитории к единому стандарту документации

**Действия:**
- Создать стандартную структуру docs/ в каждом продукте
- Подготовить module.json конфигурации для каждого репозитория
- Создать базовый контент документации (заглушки основных разделов)
- Настроить workflows уведомлений в каждом репозитории

**Репозитории для обработки:**
- `info-tech` - корпоративный контент
- `quiz` - документация Quiz Engine
- `hugo-templates` - документация Hugo Templates Framework
- `web-terminal` - документация Web Terminal
- `info-tech-cli` - документация CLI инструментов

**Стандартная структура docs/:**
```
{product}/docs/
├── module.json              # Конфигурация для сборки
├── content/
│   ├── _index.md           # Главная страница продукта
│   ├── getting-started/    # Быстрый старт
│   ├── user-guide/         # Руководство пользователя
│   ├── api/                # API документация
│   └── contributing/       # Руководство для контрибьюторов
└── .github/workflows/
    └── notify-hub.yml      # Уведомление хаба об изменениях
```

### 2.4 Настройка CI/CD автоматизации (1 день)
**Цель:** Обеспечить автоматическое обновление GitHub Pages при изменениях в docs/

**Действия:**
- Создать notify-hub workflows во всех продуктовых репозиториях
- Настроить deploy-complete-sites workflow в хаб-репозитории
- Протестировать цепочку уведомлений репозиторий → хаб → GitHub Pages
- Настроить обработку ошибок и логирование процессов сборки

**Схема автоматизации:**
```
Изменение в {product}/docs/content/
    ↓ (GitHub push event)
{product}/.github/workflows/notify-hub.yml
    ↓ (repository_dispatch)
info-tech-io.github.io/.github/workflows/deploy-complete-sites.yml
    ↓ (Hugo build + deploy)
GitHub Pages: info-tech-io.github.io
```

**Выходные артефакты:**
- Функциональные notify-hub workflows во всех репозиториях
- Работающий deploy-complete-sites workflow в хабе
- Протестированная цепочка автоматизации

## Риски и митигация

### Возможные риски
1. **Сложность интеграции:** Новые шаблоны конфликтуют с существующим framework

2. **Производительность GitHub Actions:** Длительное время сборки

3. **Ошибки в workflows:** Неработающая автоматизация
