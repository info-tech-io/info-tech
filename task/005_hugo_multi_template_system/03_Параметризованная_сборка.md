# Этап 3: Параметризованная система сборки

## Цель этапа
Создать гибкую систему параметризованной сборки с CLI интерфейсом, поддержкой всех templates, themes и components.

## Длительность
**2 дня** (продолжение Фазы 2)

## Детальные шаги выполнения

### Шаг 3.1: Разработка главного скрипта build.sh
- Создать `scripts/build.sh` с поддержкой CLI параметров
- Реализовать парсинг аргументов: `--template`, `--theme`, `--components`, `--content`, `--module`
- Добавить валидацию входных параметров
- Обеспечить обратную совместимость с hugo-base режимом

### Шаг 3.2: Система управления компонентами
- Реализовать логику включения/исключения компонентов на основе components.yml
- Создать механизм копирования/слияния статических файлов компонентов
- Добавить валидацию совместимости компонентов с template/theme
- **ВАЖНО:** Только Quiz Engine полностью работает, остальные - заготовки

### Шаг 3.3: Интеграция с системой тем
- Добавить поддержку динамического выбора темы
- Реализовать валидацию существования выбранной темы
- Обеспечить fallback на compose тему при отсутствии запрошенной
- **ВАЖНО:** Только compose тема работает, остальные - примеры

### Шаг 3.4: Система конфигурации Hugo
- Создать динамическую генерацию hugo.toml на основе template + theme + components
- Реализовать слияние базовой конфигурации template с настройками компонентов  
- Добавить override возможности для специфических модулей
- Обеспечить корректную настройку путей и зависимостей

### Шаг 3.5: Вспомогательные скрипты
- `scripts/validate-components.sh` - проверка корректности components.yml
- `scripts/test-template.sh` - тестирование конкретного template
- `scripts/theme-switcher.sh` - переключение между темами
- Создать базовый `scripts/deploy.sh` для будущей интеграции с CI/CD

## Способы верификации

### 🔍 Проверка основного функционала build.sh

#### Тест базовой сборки (default)
```bash
./scripts/build.sh --template=default --theme=compose --components=quiz-engine --content=test-content --module=test
echo $?  # Должен быть 0 (успех)
```

#### Тест minimal сборки  
```bash
./scripts/build.sh --template=minimal --theme=compose --components=navigation --content=test-content --module=test
ls output/  # Должен содержать базовые файлы Hugo
```

#### Тест валидации параметров
```bash
./scripts/build.sh --template=nonexistent 2>&1 | grep -i "error"
./scripts/build.sh --theme=nonexistent 2>&1 | grep -i "fallback"
./scripts/build.sh --help | wc -l  # > 20 строк справки
```

### 🔍 Проверка системы компонентов
```bash
# Тест валидации components.yml
./scripts/validate-components.sh templates/default/components.yml
./scripts/validate-components.sh templates/minimal/components.yml

# Проверка интеграции Quiz Engine
find output/ -path "*/quiz/*" | wc -l  # Должно быть > 0 для default template

# Проверка заглушек для планируемых компонентов  
./scripts/build.sh --components=analytics 2>&1 | grep -i "stub\|placeholder"
```

### 🔍 Проверка системы тем
```bash
# Тест существующей темы
./scripts/theme-switcher.sh compose
ls themes/compose/  # Должна существовать

# Тест fallback на несуществующую тему
./scripts/build.sh --theme=nonexistent 2>&1 | grep -i "fallback.*compose"
```

### 🔍 Проверка генерации конфигурации
```bash
# Тест генерации hugo.toml
./scripts/build.sh --template=default --theme=compose --components=quiz-engine --dry-run
cat generated-hugo.toml | grep -E "(theme|baseURL|title)"

# Валидация Hugo конфигурации
hugo config --config=generated-hugo.toml | grep -v "ERROR"
```

### 🔍 Проверка производительности
```bash

time ./scripts/build.sh --template=minimal --theme=compose --components=navigation
time ./scripts/build.sh --template=default --theme=compose --components=quiz-engine
# minimal должен быть заметно быстрее default
```

## Риски и митигация

### 🔴 Высокий риск: Сложность CLI интерфейса
**Риск:** Слишком сложный CLI может отпугнуть пользователей
**Митигация:** Хорошие defaults, понятная справка, примеры использования

### 🔴 Высокий риск: Совместимость с Hugo
**Риск:** Динамически генерируемые конфигурации могут нарушить работу Hugo  
**Митигация:** Валидация всех генерируемых конфигураций, тесты с реальными примерами

### 🟡 Средний риск: Производительность системы сборки
**Риск:** Дополнительные проверки могут замедлить процесс сборки
**Митигация:** Профилирование скриптов, оптимизация критических путей

### 🟡 Средний риск: Complexity debt от заготовок
**Риск:** Создание заглушек для несуществующих компонентов может запутать систему
**Митигация:** Четкое документирование статуса, простые и понятные заглушки
