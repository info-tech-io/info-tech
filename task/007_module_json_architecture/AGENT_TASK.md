# Задача 007: Реализация децентрализованной архитектуры с module.json

## Общая концепция и цель

Реализация децентрализованной архитектуры управления образовательными модулями через введение файлов `module.json` в каждый репозиторий `mod_*`. Данное изменение направлено на упрощение процесса подключения/отключения новых учебных модулей и переход к self-contained архитектуре, где каждый модуль содержит всю необходимую метаинформацию о себе.

## Предыстория и мотивация

### Текущая архитектура (централизованная)
В настоящее время вся информация о модулях хранится в центральном файле `modules.json` в репозитории `infotecha`. Это создает следующие проблемы:

1. **Узкое место при добавлении модулей**: Каждый новый модуль требует изменения центрального файла
2. **Версионирование конфигурации**: Изменения в конфигурации модуля не связаны с версией самого модуля
3. **Сложность onboarding**: Создатели модулей должны знать структуру центрального файла
4. **Риск конфликтов**: Одновременное добавление нескольких модулей может привести к конфликтам merge
5. **Отсутствие автономности**: Команды модулей зависят от центральной команды для изменений

### Целевая архитектура (децентрализованная)
Каждый модуль содержит файл `module.json` в корневой директории с полной информацией о себе:
- Метаданные модуля (название, описание, автор)
- Конфигурация сборки (шаблон, тема, компоненты)
- URL-адреса и технические параметры
- Теги, сложность, время изучения

## Структура module.json

### Полная схема файла:
```json
{
  "schema_version": "1.0",
  "name": "linux-base",
  "title": "Основы Linux",
  "description": "Полный курс для начинающих пользователей Linux",
  "version": "1.0.0",
  "type": "educational",

  "deployment": {
    "subdomain": "linux-base",
    "repository": "mod_linux_base",
    "build_system": "hugo-base"
  },

  "hugo_config": {
    "template": "default",
    "theme": "compose",
    "components": ["quiz-engine"],
    "hugo_version": "0.148.0"
  },

  "metadata": {
    "author": "InfoTech.io Team",
    "maintainer": "info-tech-io",
    "license": "MIT",
    "difficulty": "beginner",
    "estimated_time": "40 hours",
    "language": "ru",
    "tags": ["linux", "command-line", "system-administration", "terminal"]
  },

  "urls": {
    "production": "https://linux-base.infotecha.ru",
    "repository": "https://github.com/info-tech-io/mod_linux_base",
    "issues": "https://github.com/info-tech-io/mod_linux_base/issues"
  },

  "status": {
    "lifecycle": "stable",
    "last_updated": "2025-09-20",
    "content_complete": true,
    "testing_complete": true
  }
}
```

### Поля для будущей совместимости с hugo-templates:
- `hugo_config.template` - тип шаблона для сборки
- `hugo_config.components` - список используемых компонентов
- `hugo_config.theme` - выбранная тема

## Миграционная стратегия

### Принцип: Постепенная миграция с обратной совместимостью

**Этап 1**: Добавление module.json файлов (не ломая существующую систему)
**Этап 2**: Обновление центральной логики для чтения из обоих источников
**Этап 3**: Переключение на module.json как primary source of truth
**Этап 4**: Deprecation центрального modules.json
**Этап 5**: Полное удаление legacy системы

### Backwards Compatibility Strategy:
1. Центральный modules.json остается как fallback
2. module.json имеет приоритет, если существует
3. Валидация обеих систем параллельно
4. Постепенная миграция без downtime

## Преимущества реализации

### Для разработчиков модулей:
- **Автономность**: Изменения не требуют доступа к центральному репозиторию
- **Простота**: Вся конфигурация в одном месте
- **Версионирование**: Конфигурация связана с версией модуля
- **Валидация**: Автоматическая проверка корректности конфигурации

### Для платформы:
- **Масштабируемость**: Легкое добавление новых модулей
- **Автоматизация**: Автоматическое обнаружение изменений
- **Надежность**: Отсутствие единой точки отказа
- **Мониторинг**: Детальная информация о статусе каждого модуля

### Для пользователей:
- **Актуальность**: Информация о модуле всегда синхронизирована
- **Прозрачность**: Полная информация о модуле доступна в его репозитории
- **Качество**: Улучшенная валидация и контроль качества

## Риски и ограничения

### Технические риски:
1. **Сложность миграции**: Необходимо обеспечить zero-downtime переход
2. **Синхронизация**: Возможны расхождения между центральным и локальными файлами
3. **Валидация**: Необходимо проверять корректность всех module.json файлов
4. **Performance**: Сканирование множества репозиториев может быть медленным

### Операционные риски:
1. **Обучение команды**: Необходимо обновить процедуры и документацию
2. **Rollback complexity**: Откат к старой системе может быть сложным
3. **Monitoring**: Необходимо отслеживать консистентность данных

### Митигация рисков:
- Постепенная миграция с тестированием на каждом этапе
- Comprehensive валидация и автоматические проверки
- Детальное логирование и мониторинг процессов
- Процедуры быстрого отката в случае проблем
