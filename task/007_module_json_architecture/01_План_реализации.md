# План реализации: Децентрализованная архитектура с module.json

## Стратегия реализации: "Zero-Downtime Migration"

### Основные принципы:
1. **Backwards Compatibility First**: Никаких поломок существующей системы
2. **Gradual Migration**: Постепенный переход с валидацией на каждом этапе
3. **Safety Nets**: Возможность быстрого отката на каждом этапе
4. **Validation Heavy**: Комплексная валидация всех изменений

### Этап 2: Система чтения module.json (2 дня)
**Цель**: Добавить функциональность чтения module.json без ломки существующего

#### 2.1. Создание утилиты сканирования модулей (1 день)
**Задачи:**
- Создать скрипт сканирования всех mod_* репозиториев
- Реализовать чтение module.json через GitHub API
- Добавить кеширование для производительности
- Создать fallback на центральный modules.json

**Новый файл: infotecha/scripts/scan-modules.js**
```javascript
// Основная функциональность:
// 1. Сканирование всех репозиториев info-tech-io/mod_*
// 2. Чтение module.json из каждого репозитория
// 3. Валидация против JSON Schema
// 4. Merge с центральным modules.json (приоритет у module.json)
// 5. Генерация итогового modules.json
```

#### 2.2. Интеграция с GitHub Actions (1 день)
**Задачи:**
- Обновить .github/workflows/module-updated.yml
- Добавить автоматическое сканирование при изменениях
- Сохранить обратную совместимость
- Добавить валидацию изменений

**Обновленный workflow:**
```yaml
name: Module Updated v2
on:
  repository_dispatch:
    types: [module-updated, module-updated-v2]

jobs:
  build-module:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout infotecha
        uses: actions/checkout@v3

      - name: Scan for module.json
        run: |
          if node scripts/scan-modules.js --module ${{ github.event.client_payload.module }} --validate; then
            echo "MODULE_CONFIG_SOURCE=module.json" >> $GITHUB_ENV
          else
            echo "MODULE_CONFIG_SOURCE=central" >> $GITHUB_ENV
          fi

      - name: Get module config
        run: |
          if [ "$MODULE_CONFIG_SOURCE" = "module.json" ]; then
            # Читаем из module.json репозитория
            MODULE_CONFIG=$(node scripts/scan-modules.js --module ${{ github.event.client_payload.module }} --output json)
          else
            # Fallback на центральный modules.json
            MODULE_CONFIG=$(jq ".modules[] | select(.repository == \"${{ github.event.client_payload.module }}\")" modules.json)
          fi
          echo "MODULE_CONFIG=$MODULE_CONFIG" >> $GITHUB_ENV

      # Остальная логика сборки остается той же
```

## Риски и митигация

### Критические риски:
1. **Поломка существующих модулей**
   - *Митигация*: Extensive тестирование + rollback plan
2. **Производительность GitHub API**
   - *Митигация*: Кеширование + rate limiting
3. **Синхронизация данных**
   - *Митигация*: Валидация на всех уровнях

### План отката:
На любом этапе можно вернуться к полностью централизованной архитектуре:
1. Отключить чтение module.json в workflows
2. Восстановить полный центральный modules.json
3. Откатить изменения в GitHub Actions
4. Проверить работоспособность всех модулей
